✓ 1 documento root · N sub-perfiles · 1 sola persona
  • 1 email · 1 foto · 1 documento · datos base unificados
  • Multi-rol · una persona puede tener 1+ roles
  • Sub-perfiles solo si aplican (rol vendedor → datos laborales)
  • Clientes/Proveedores quedan separados (entidades externas)
Regla canon · cuándo es User vs cuándo es entidad externa
→ Modelar como User (persona INTERNA del negocio)
  • Operan el sistema (tienen login)
  • Trabajan o son dueños del negocio
  • Roles posibles: admin · gerente · vendedor · comprador · almacenero · finanzas · supervisor · invitado · socio · colaborador interno
→ Modelar como entidad externa (contraparte comercial)
  • NO operan el sistema (sin login)
  • Contraparte comercial · compran o venden
  • Colecciones: /clientes · /proveedores · /colaboradores externos
  • Vinculación opcional con User si en el futuro adquieren login

ACTO 3 · Onboarding adaptado al nuevo modelo

Cómo el onboarding del módulo Inversionistas se simplifica aún más con multi-rol.

📍 ESTADO C1 · Inversionistas · setup automático en 1 click cuando el user es admin
Setup express · Inversionistas

Convertirte en socio · 1 click

Detectamos que sos admin del sistema. Si también sos el único socio del negocio, te agregamos el rol socio y completamos la sub-vista de datos de socio.

Cambios que vamos a hacer:
  • Agregar rol "socio" a tu usuario (mantienes el resto)
  • Crear sub-perfil /users/{tu uid}/datosSocio · 100% · hoy
  • Vincular automáticamente tus TCs personales (si marcás abajo)

Decisiones canon · Modelo de personas v5.2

D1 · Persona = UserProfile con multi-rol
  • role: UserRole → roles: UserRole[]
  • Permisos se calculan como union de los roles
  • Nuevo rol 'socio' agregado a UserRole
  • Sidebar y rules usan 'admin' in roles
D2 · Sub-perfiles solo si aplican
  • /users/{uid}/private/datosLaborales si tiene rol planilla
  • /users/{uid}/private/datosSocio si tiene rol socio
  • Tabs dinámicas en modal · aparecen según roles
  • Migración: /empleados y /socios → sub-perfiles
D3 · Clientes/Proveedores quedan separados
  • Son contrapartes comerciales · NO usan el sistema
  • /clientes y /proveedores siguen como entidades propias
  • Vinculación opcional a User vía userId (si llegan a tener login)
D4 · Ficha 360 por persona
  • Página propia · /usuarios/{uid}/ficha
  • 4 cards: permisos · planilla · socio · actividad
  • Resumen de actividad últimos 30 días
D5 · Maestros · Socios desaparece
  • El catálogo /socios separado se elimina
  • Inversionistas lee /users con rol socio
  • Sub-tab Maestros · Socios queda en limbo · se elimina del sidebar
D6 · Onboarding más simple
  • Setup en 1 click (admin único · 80% PyMEs)
  • "Tengo varios socios" → lleva a Usuarios para crearlos
  • Sub-perfil datosSocio se crea automáticamente
D7 · Participación = Cash + TC + VALOR · NO solo dinero
  • 3 tipos de participación: cash puro · valor puro · mixta (lo más común)
  • 7 tipos de valor: know-how · gestión/CEO · networking/cartera · marca personal · idea original/IP · tiempo (sweat equity) · activos no monetarios · otro
  • % participación NO es proporcional al cash · puede haber socios con cero cash y 30% (silent partners que traen cartera/marca/etc)
  • Valuación del valor · opcional · si el contador o acuerdo de socios lo valuó (ej. "cartera vale S/200k")
  • Vesting · opcional · materialización con el tiempo (lineal · cliff · inmediato)
  • Capital Comprometido (KPI principal) sigue siendo cash + TC (lo medible en dinero) · el valor aporta a la composición del cap table pero no al patrimonio contable
  • Vista en Inversionistas: KPI strip de cap table del negocio (cash · TC · valor) + breakdown por socio con 3 chips de aporte y tags de tipos de valor

  • Sweep · ~40-60 lugares con userProfile.role === 'X'
  • Sidebar · cambiar gating
  • Migración 1x · convertir role: "admin" → roles: ["admin"] · script node
  • Forms /usuarios · multi-select de roles
2 Fase 2 · Sub-perfiles + Ficha 360 (~6-8 horas)
  • Tipos · DatosLaborales · DatosSocio
  • Services · datosLaborales.service · datosSocio.service
  • Modal /usuarios extendido · tabs dinámicas según roles
  • Ficha 360 nueva página · /usuarios/{uid}/ficha
  • Migración · /empleados → sub-perfil datosLaborales · /socios → datosSocio
  • Eliminar colecciones /empleados y /socios después de migrar
3 Fase 3 · Adaptar Inversionistas + Planilla + Onboarding (~4-6 horas)
  • Planilla · lee /users con rol planilla en vez de /empleados
  • Inversionistas · lee /users con rol socio · sub-tab Socios en Maestros queda eliminado
  • Onboarding wizard · ahora simple "agregar rol socio + crear datosSocio"
  • Modales aporte/retiro · usan userId en vez de socioId
  • TC personal · vinculada vía titularEntidadTipo='user' titularEntidadId=uid (en lugar de 'socio')
  • Eliminar todo el código de socio.service.ts + socioStore + SociosMaestros.tsx
Total estimado: ~16-22 horas trabajo focalizado · 3 sesiones grandes dedicadas.
