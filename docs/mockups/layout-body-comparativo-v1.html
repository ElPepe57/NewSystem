¿Es necesaria la desviación A vs B?
✓ Sí es legítima (macro)
  • El sidebar (Layout A) existe porque Finanzas y Gastos tienen contexto operativo urgente (qué vence, qué cobrar, pool USD) que el usuario necesita ver mientras opera la lista, sin scrollear.
  • Contabilidad muestra estados financieros anchos (Balance, P&L) que necesitan todo el ancho — un sidebar les robaría espacio crítico.
  • Forzar un layout único degradaría a uno de los dos grupos.
✗ NO es legítima (micro)
  • Que Gastos use max-w-7xl y Finanzas full — siendo ambos Layout A — no tiene razón.
  • Que Contabilidad sea px-6 sin max-w mientras Usuarios/Planilla/Inversionistas son max-w-6xl p-3 sm:p-4 md:p-6 — sin razón.
  • El orden del Resumen (banner→viz→insights→cross-links→alertas) está documentado en Planilla pero Gastos no lo respeta.
Veredicto

La desviación A vs B sí es necesaria — responde a la naturaleza del contenido (operativo-con-contexto vs financiero-ancho). Lo que no es aceptable es la inconsistencia micro (max-w, padding, orden de Resumen) dentro de cada grupo. La alineación correcta NO es "un layout para todos", sino 2 layouts canónicos bien definidos + micro-detalles idénticos dentro de cada uno.

Alternativas de alineación · de menos a más ambiciosa
Opción 1 · Normalizar micro-detalles (mínima · bajo riesgo)
RECOMENDADA como base
Mantener A y B, pero alinear: contenedor único max-w-6xl mx-auto + padding único p-3 sm:p-4 md:p-6 en TODOS. Layout A conserva su grid+sidebar dentro de ese contenedor.
+ Bajo riesgo · cambios de clase puntuales – No unifica el layout macro (sigue A y B)
Toca: Contabilidad (add max-w-6xl + padding) · Finanzas (add max-w-6xl) · Gastos (7xl→6xl + padding).
Opción 2 · Componente <HubBody> con slot opcional (ideal técnica)
Un solo componente que unifica el CÓDIGO; el sidebar es un slot opcional:
<HubBody aside={<Urgentes/>}>   {/* Layout A si pasás aside */}
  {contenido main}
</HubBody>

<HubBody>                          {/* Layout B si NO pasás aside */}
  {contenido full-width}
</HubBody>
El componente resuelve internamente: contenedor, max-w, padding, grid vs full-width, y el responsive. Imposible desviarse a futuro.
+ 1 fuente de verdad · cero desviaciones futuras · A y B salen del mismo componente – Refactor de los 6 módulos (mediano)
Opción 3 · Forzar 1 layout único para todos (no recomendada)
Convertir todo a full-width (mover sidebars a cards en el flujo) o todo a grid+sidebar.
+ Consistencia visual absoluta – Degrada un grupo: o Finanzas/Gastos pierden el panel de urgencias, o Contabilidad pierde ancho para el Balance. Va contra la función.
Recomendación

Declarar 2 layouts canónicos de body (A · grid+sidebar · para dashboards operativos con contexto · y B · full-width · para contenido financiero/directorio ancho) con regla clara de cuándo usar cada uno, y normalizar los micro-detalles (max-w-6xl + p-3 sm:p-4 md:p-6 + orden de Resumen §A→§F) idénticos dentro de cada uno.

Camino sugerido: Opción 1 ahora (normalizar · bajo riesgo) → registrar el canon de 2 layouts en CLAUDE.md → Opción 2 (componente <HubBody>) como meta de mediano plazo para blindar la consistencia.
