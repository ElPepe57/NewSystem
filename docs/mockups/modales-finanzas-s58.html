1 · Filosofía del sistema

Crear datos no debería ser un trámite.

Las grandes empresas (Stripe, Mercury, Linear) tratan los formularios como conversaciones — no como obstáculos. El usuario llega con una intención (registrar un movimiento, hacer una conversión) y el sistema lo guía con defaults inteligentes, validación en vivo, atajos y feedback inmediato. Si todo va bien, el usuario sale en menos de 8 segundos.

PRINCIPIO 1
Default is friend
Cada campo arranca con el valor más probable. Fecha = hoy, TC = del día, cuenta = la más usada.
PRINCIPIO 2
Validate as you go
El error aparece en el campo, en el momento, no al hacer submit. Verde cuando OK, rojo cuando no.
PRINCIPIO 3
Optimistic by default
Al hacer submit, el modal se cierra al instante. Si algo falla, banner discreto con [Deshacer].
PRINCIPIO 4
Don't lose work
Si cierras por accidente, tu borrador se guarda. Al volver, "¿Continuar borrador del miércoles?".
PRINCIPIO 5
Keyboard first
TAB navega, Esc cierra, ⌘↵ guarda. Mostrados en el footer del modal.
PRINCIPIO 6
Calm color, loud action
Fondo blanco. Botones tonales. La acción primaria sobresale por contraste, no por saturación.

HEADER · sticky
  • • Icon tonal (40×40, bg-teal-50 border-teal-200) — identidad inmediata
  • • Título (text-base font-bold) — qué se está creando
  • • Breadcrumb (text-[11px] text-slate-500) — contexto: "Cash flow · Movimiento"
  • • Close (sólo X, no "Cancelar"; cancelar va en el footer)
BODY · scrollable
  • • Sin secciones de colores — la jerarquía la da tipografía y espaciado
  • • Labels uppercase tracking-wider text-[11px] (Stripe Atlas)
  • • Inputs height 40px, border-slate-300, focus-ring-teal-500
  • • Validación inline debajo de cada input
  • • max-h-[80vh] + overflow-auto si hay muchos campos
FOOTER · sticky
  • • Atajos teclado visibles a la izquierda (kbd-style)
  • • Cancelar = ghost (no llama la atención)
  • • Acción primaria = primary-soft (no saturado)
  • • Si hay auto-save, indicador "Guardado hace 2s" en lugar de atajos

4 · Smart defaults · "el sistema te ayuda"

Cada vez que abres un modal, los campos llegan PRE-LLENADOS con la mejor adivinanza del sistema. El usuario solo cambia lo distinto.

Tipos de hint

¿Por qué es valioso?
  • • Reduce tiempo de creación de ~30s a ~8s
  • • Reduce errores: usar TC del día > tipearlo de memoria
  • • Reduce friction: menos campos vacíos = menos ansiedad
  • • El usuario aprende qué se considera "estándar"

Validación · 3 momentos
1. Mientras escribes
Solo errores graves (ej: "monto debe ser mayor a 0"). No molestar al usuario que aún está escribiendo.
2. Al salir del campo (blur)
Validación completa del campo: formato, longitud, unicidad. Mensaje rojo debajo si falla.
3. Al hacer submit
Si quedan errores: banner rojo arriba con sumario + scroll al primer campo en error.
Empty state dentro del modal
Sin resultados
No encontramos ventas con ese número.
Cuando un combobox o búsqueda no encuentra nada, ofrecer crear el recurso ahí mismo (mini-modal anidado).
Loading · skeleton, no spinner
El skeleton respeta el layout final. El usuario no ve un spinner girando — ve la estructura llenándose.
