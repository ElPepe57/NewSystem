Manifesto · 8 reglas canónicas

Cada vista del módulo Ventas DEBE cumplir las 8. Cualquier desviación requiere autorización explícita y se documenta como excepción declarada.

R1
Canvas protagonista

El mapa · gráfica · matriz ocupa 65-75%

El elemento visual principal NUNCA es una tabla o lista. Es siempre un canvas explorable (mapa · heatmap · gráfico). Paleta sutil para no competir con cards.

Aplicación: Vista 1 (mapa Perú) · Vista 5 (Lima) · Vista 8 (cards anchored). Ratios: 3:1 grid (canvas:drawer).
R2
Cards anchored

Información primaria como cards flotantes

NO sidebars con listas planas. NO tablas crudas. Cada dato relevante es una card anclada a coordenadas/ejes/tiempo. Cards persistentes (no tooltips).

Componente: <AnchoredCard> con prop anchorTo={{x,y}}.
R3
2 modos siempre

Expandida (1-3) · Compacta (5-8) · Cluster (≥3)

Densidad informativa sin saturación. Card expandida tiene foto + métricas + sparkline + CTAs. Card compacta tiene 1 línea de info esencial. Cluster cuando se solapan.

Reglas: max 3 expandidas por canvas · max 8 compactas · cluster a partir de 3 cercanas.
R4
Header gradient

Purple-fuchsia direccional · señal de "info clave"

Toda card expandida lleva header con gradient. Diagonal 135deg. Color from-purple-600 to-fuchsia-600. Variante para top: to-purple-900.

Tokens: --gradient-card-default · --gradient-card-top.
R5
Sparkline interna

Mini-tendencia DENTRO de cada card · NO charts separados

Donde haya tendencia temporal, va sparkline EN la card. Adiós a donut charts gigantes y barras separadas. Stripe Radar pattern.

Componente: <SparklineMini data={'{[]}'} /> · 100×24px · gradient fill opacity 0.3.
R6
Tabular-nums obsesivo

Banking-grade · todo monto y métrica

Cualquier número (S/ · unidades · fechas · IDs · porcentajes) usa tabular-nums. Stripe lo hace en TODO el dashboard. Diferencia entre amateur y pro.

Default global: font-variant-numeric: tabular-nums en class utility.
R7
Drawer reorderable

Stack lateral 25% · drag&drop · NO tablas

Si hay listado complementario al canvas, va como stack de cards mini en drawer derecho · drag handle ☰ · usuario prioriza manualmente. Pattern Linear.

Componente: <StackDrawer items={'{[]}'} reorderable />.
R8
Story mode opcional

Botón ▶ Tour · 3-5 escenas guiadas

Cualquier dashboard puede convertirse en narrativa. Útil para onboarding · presentaciones · auditorías. Pattern Tableau Spatial.

Componente: <StoryMode escenas={'{[]}'} /> con CTA flotante violet.

Cómo se aterrizan las 8 vistas previas en el style canónico

Cada vista previa tenía su propio "tono". Aplicando las 8 reglas + 5 componentes, todas respiran lo mismo. Ejemplos comparativos antes/después.

Vista 1 → canónico

Mapa Perú nacional

❌ Antes
Mapa SVG + sidebar de lista con 5 regiones top + insights stack. 3 áreas separadas. Estilo dashboard genérico.
✓ Aplicando R1+R2+R3
Mapa Perú al 75% + 3 AnchoredCard expandidas sobre Lima/Arequipa/Callao + 4 compactas sobre regiones secundarias + StackDrawer derecho. Mismo lenguaje que Vista 8.
Vista 2 → canónico

Heatmap día × hora

❌ Antes
Grid GitHub-style 7×24 puro + insights al pie. Estilo calendar contributions. Otro lenguaje visual que no encaja con cards anchored.
✓ Aplicando R2+R3+R5
Heatmap como canvas + 3 AnchoredCard sobre los picos (Sáb 14h-16h · Vie 19h-21h · Mar 20h-21h) con sparkline embedded mostrando últimas 4 semanas. Cards anchored a celdas del grid.
Vista 3 → canónico

Recurrencia clientes

❌ Antes
3 cards separadas (donut + top10 + cohorts). BI dashboard clásico. El donut compite con las otras visualizaciones.
✓ Aplicando R5+R7
Donut se elimina · reemplazado por StackDrawer con 3 secciones (VIP · Regulares · Una compra). Top 10 LTV son AnchoredCard sobre línea temporal. Cohorts como sparklines en cards. Sin charts separados.
Vista 4 → canónico

Drill-down regional

❌ Antes
3 columnas (distritos con barras + top productos + sugerencias). Estilo reporte management. Las barras de progreso compiten con cards.
✓ Aplicando R2+R7
Mapa Lima al 75% + AnchoredCards sobre top distritos con producto destacado + sparkline. Las "sugerencias" se convierten en cards expandidas con CTAs directos · NO listas.
Vista 5 → canónico

Zoom Lima Z1

❌ Antes
Mapa Lima + sidebar contextual con KPIs + áreas blancas + acciones. Estilo Google Maps + dashboard. La sidebar "lista" rompe el flujo visual.
✓ Aplicando R1+R2+R7
Es prácticamente Vista 8. Mapa Lima + AnchoredCards sobre top 6 distritos + StackDrawer con resto + ChipBar de capas. Zoom Z2 desde card expandida.
Vistas 6+7 → canónico

Heatmap matriz Producto×Región/Tiempo

❌ Antes
Tabla densa coloreada estilo Excel/Tableau. Funcional pero árida. Las celdas son texto puro · no hay personalidad visual.
✓ Híbrido R2+R5
Matriz se mantiene como canvas legítimo (es el patrón natural del cruce) · pero las celdas con valor alto muestran mini-AnchoredCard con foto SKU + sparkline al hover prolongado · drawer derecho con stack reorderable de SKUs. Mejor de ambos mundos.
⚠️
Excepción legítima · Heatmap matriz mantiene su forma

El cruce Producto × Región es naturalmente tabular · forzar AnchoredCards en una matriz 20×11 sería contraproducente. Solución híbrida: matriz como canvas + cards como tooltips persistentes solo en celdas top. Mantiene la legibilidad de la matriz y agrega personalidad del style canónico. Documentada como excepción declarada D-VENTAS-CANONICO-3.

Decisiones canónicas declaradas

D-VENTAS-CANONICO-1
Vista 8 (cards anchored) es ANCLA del style language del módulo Ventas. Cualquier vista futura · cualquier rediseño · cualquier migración respeta las 8 reglas del manifesto. Las vistas previas (1-7) se aterrizan progresivamente al canónico.
D-VENTAS-CANONICO-2
5 componentes canónicos como librería compartida: AnchoredCard (expandida + compacta) · StackDrawer · ChipBar · Cluster · SparklineMini. Implementar 1 vez · usar en todo. Vivirán en src/components/ventas/canonico/.
D-VENTAS-CANONICO-3
Excepción declarada · Heatmap matriz. El cruce Producto × Región/Tiempo mantiene su forma tabular como canvas legítimo (forzar AnchoredCards 20×11 = contraproducente). Solución híbrida: matriz + cards como tooltips persistentes en celdas top.
D-VENTAS-CANONICO-4
Tabular-nums obsesivo · default global. Class utility en CSS root: .tabular-nums auto-aplicada a montos S/ · unidades · porcentajes · IDs · fechas. Banking-grade quality. Diferencia entre amateur y pro.
D-VENTAS-CANONICO-5
Paleta primaria fija · purple-fuchsia gradient. Todo el módulo Ventas respira el mismo color. Gradients direccionales 135deg. Variantes: hero (purple→fuchsia) · top (purple-700→purple-900) · alterno (fuchsia→pink). Los acentos semánticos (emerald · rose · amber) NUNCA cambian de código.
D-VENTAS-CANONICO-6
Plan de migración progresiva · 4 fases:
  • F1 (1 sesión): Implementar 5 componentes canónicos en /src/components/ventas/canonico/
  • F2 (1 sesión): Refactorizar Vista 8 (cards anchored) usando los componentes · valida la librería
  • F3 (2 sesiones): Migrar Vistas 1, 4, 5 (las más usadas) al canónico
  • F4 (1 sesión): Migrar Vistas 2, 3, 6, 7 (heatmap matriz con excepción D-CANONICO-3)
Total estimado: ~5 sesiones. Sin breaking · cada vista migrada queda funcional.
D-VENTAS-CANONICO-7
Replicar el patrón a otros módulos del sistema. El style canónico definido para Ventas puede aplicarse a Inventario · Tesorería · Envíos. Cada módulo mantendría su color primario (Inventario=teal · Tesorería=emerald · Envíos=indigo) pero respetaría las 8 reglas estructurales. Documentado como TAREA-DESIGN-SYSTEM-GLOBAL · post-S58f.
