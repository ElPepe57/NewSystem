🌳 ¿Cómo se crea un envío?

Los envíos se crean de dos formas: automáticamente (vienen de otro documento como OC/Venta/Devolución) o manualmente (movimiento logístico independiente desde /envios).

🤖

Ruta 1 — Automática · Sin preguntas al usuario

El sistema usa los datos que ya fueron capturados en otro módulo. El usuario no interactúa con el módulo de envíos para crearlos — el envío aparece como consecuencia de otra acción.

Desde una OC (Compras)
Al confirmar OC, confirmarOC() lee orden.modoEntregaDetallado y crea el envío automáticamente:
ddp_directo → Caso B
via_viajero / via_courier → Caso A
recojo_en_origen → Caso D
La OC ya trae: proveedor, casilla destino, colaborador transporte, D1/D2. No se pregunta nada.
Desde una Venta
Al confirmar venta, se crea envío de despacho hacia el cliente.
Venta confirmada → Caso F
La venta ya trae: almacén origen, cliente, dirección entrega, courier. El envío F se crea silencioso.
Desde una Devolución
Al abrir devolución en Ventas, se crea envío de retorno.
Devolución abierta → Caso G
Hereda: cliente origen, venta original, almacén destino, política de flete.
✓ Principio: si la info ya está en la OC/Venta/Devolución, no se vuelve a preguntar. El sistema infiere el tipo (A/B/D/F/G) y hereda todos los campos automáticamente. El usuario ve el envío aparecer en /envios como resultado, no lo crea desde ahí.
✋

Ruta 2 — Manual · El usuario responde preguntas simples

Solo para movimientos logísticos independientes — aquellos que no nacen de una OC/Venta/Devolución. El usuario abre /envios, click "Nuevo envío", y responde un mini-wizard:

📌 Principios del modelo de sub-envíos
  • Los sub-envíos son opcionales: un envío puede recibirse entero sin fraccionar (ej. OC pequeña en 1 paquete). La sección "Tandas" solo aparece si existen sub-envíos.
  • Se pueden crear en cualquier estado del T1 (excepto recibida_completa / cancelada). No hay que regresar a borrador.
  • El pipeline del T1 se actualiza automáticamente según los sub-envíos: primer "en tránsito" activa el T1, primer "entregado" dispara recepción parcial.
  • Editabilidad: tanda pendiente o en_transito = editable (unidades, tracking, fecha). Tanda entregado o entregado_parcial = solo lectura.
  • Modo mixto: puedes crear tandas prospectivas (antes del despacho) o reactivas (al recibir). Ambas coexisten sin fricción.
  • Reemplazo físico (D-16): si una tanda llega incompleta, se abre reclamo al proveedor. Si acepta reemplazar, se crea una nueva tanda tipo=reemplazo en el mismo envío T1, con CTRU preservado de la unidad original. Si no asume, se declara merma.

🎯 Picking priorizado — detalle del algoritmo

Como el sistema sugiere que unidades incluir en un envio T2

Orden de prioridad

1
Pre-vendidas (reservada_para = cotizacion)
Cotizaciones con adelanto PAGADO. El sistema las recomienda siempre al principio del picking.
💡 Campo Unidad.reservadaPara (ya existe)
2
Stock antiguo (FIFO por fechaRecepcion)
Entre unidades no reservadas, priorizar las que llegaron primero a la casilla.
3
Stock reciente
El resto, ordenado por fechaRecepcion descendente.

Override del usuario

El sistema NO bloquea al usuario si decide no incluir pre-vendidas. Pero muestra advertencias:
