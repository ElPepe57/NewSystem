Hub Kit · la spec para construir el shell canónico (L5)

Esta es la spec de implementación de los 6 building blocks. A diferencia del blueprint conceptual (hub-design-system-v1), aquí el kit está ensamblado con contenido real, mostrando Layout A y B, la herencia de color por grupo, los estados del body y el mobile 375px. Sirve de referencia pixel-perfect para escribir HubShell / HubTopBar / HubHeader / HubKpiStrip / HubTabs / HubBody.

Radios (tokens): shell 2xl · cards xl · controles lg · pills full. Hijo nunca más redondo que el padre.
Color: chrome = color del grupo (heredado · grupoColor.ts) · datos = paleta semántica fija.
Sombra: shell shadow-sm (reposo) · overlays shadow-lg. El body va dentro del shell (1 recuadro).

Reglas de implementación del Hub Kit (en orden)

El orden §0→§7 es el orden de implementar/revisar un módulo. Cada regla está canonizada en CLAUDE.md (§A-§G). Si el módulo no cumple una de estas reglas, NO está terminado. Cambiar una regla requiere aprobación del usuario.

0Ubicación → color (no se elige)

El módulo se ubica en un grupo del sidebar y HEREDA su color de grupoColor.ts: F&C=teal · Equipo=violet · Comercial=blue · Inventario=orange · Análisis=indigo · Administración=slate. Prohibido inventar color.

1Anatomía (orden fijo · 1 recuadro)

<HubShell> → HubTopBar → HubHeader → HubKpiStrip → HubTabs → HubBody. El body va DENTRO del shell (UN recuadro continuo · nunca cards sueltas afuera). 1ª tab SIEMPRE Resumen (dashboard ejecutivo).

2Layout + contenedor

Layout A (con aside · contexto persistente) = md:grid-cols-3 · main col-span-2 + aside col-span-1 → ratio 2:1 (aside apila en mobile). Layout B (sin aside) = full-width. SIEMPRE max-w-6xl mx-auto · body bg-slate-50/30 p-3 sm:p-4 md:p-6.

3Color por FUNCIÓN (2 sistemas · NUNCA se mezclan)
  • Chrome (¿dónde estoy?) → color del GRUPO: tab activa · chip rol · primary CTA · ícono header · focus rings.
  • Dato (¿qué significa?) → paleta SEMÁNTICA fija: amber=dinero · rose=urgencia · emerald=positivo · indigo=fijo · sky=parcial · violet=capital · slate=neutral.
  • El color del grupo NUNCA pinta datos · el semántico NUNCA pinta chrome.
4Tipografía
  • Familia = Inter · única para toda la app (heredada · tailwind.config.fontFamily.sans).
  • Escala por rol: h1 text-2xl bold · métrica KPI text-2xl bold tabular-nums · título text-base/lg semibold · subtítulo/tab text-[13px] · cuerpo text-[12px] · meta/breadcrumb/delta text-[11px] · label text-[10px] bold upper · badge text-[9px]. Pocos tamaños = orden · tabular-nums SIEMPRE en números.
  • Color del texto = por función: estructura→slate (900 título/700/600 cuerpo/500 meta/400 muted) · dato→semántico · chrome→grupo.
  • Dentro de un dato, jerarquía de tono: -700 label / -900 valor / -400 decimal-atenuado (F7). El delta/sub-texto HEREDA el color de la card (card monocromática) · la dirección/tendencia va en el ÍCONO ↗↘—, NUNCA en otro color de texto.
5Componentes + formularios

Ensamblar con el DS (HubCard · FiltrosBar · SmartSearch · FormModalV2 · Button 3-tier: primary=color-grupo / config=indigo / neutral=slate). Todo form/wizard de creación: Borrador + Descartar. Iconos lucide · cero emojis en el chrome.

6Pixel-perfect + cobertura

M1: copy-paste LITERAL de las clases del mockup (NO reescribir de memoria). Cobertura TOTAL (shell + tabs + cards + modales + banners + empty/loading/error) + mobile 375px desde el inicio.

7Validación (antes de cerrar)

npm run build (tsc -b · EXIT 0) · screenshot M4 side-by-side vs este kit · sin errores de consola · dead code eliminado · commit + push.

Refleja las decisiones hasta 2026-06-01 (incl. Layout A 2:1 · tipografía Inter · color por función · delta monocromático). Canonizado en CLAUDE.md §A-§G · valores calibrados del kit: KPI card p-4/mb-2/gap-3 · tabs px-3 py-2.5 text-[13px] · header py-4 + ícono sólido from-{c}-500→700.
