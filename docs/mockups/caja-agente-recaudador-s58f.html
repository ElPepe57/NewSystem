La caja NO se opera desde aquí. Las transacciones se inician desde donde naturalmente ocurren (Ventas, Envíos, etc.) usando el componente transversal PagoUnificadoForm.

⚡ El sistema detecta automáticamente cuando el método de pago apunta a esta caja recaudadora · sin que el usuario tenga que recordarlo.

Vista 2 · Cómo se crea la caja recaudadora · lógica del sistema

Tu pregunta clave: "¿cómo creo el yape de GK en el PagoUnificadoForm? ¿Cómo entiende el sistema que tiene que crear una caja?". Aquí está la cadena conceptual + 3 caminos posibles.

Cadena conceptual · 3 niveles

1
Ficha del proveedor en Red Logística

GK Xpress tiene un campo datosBancarios[] donde registras sus yapes, cuentas, etc. Son datos pasivos · sin saldo trackeado · solo para que sepas dónde le pagas.

↓ datos pasivos disponibles
2
Detección desde PagoUnificadoForm

Cuando el usuario tipea/elige un yape al registrar cobro, el sistema busca ese número en datosBancarios[] de TODOS los proveedores. Si lo encuentra · sabe a quién pertenece · sugiere convertirlo en caja.

↓ "este yape es de GK"
3
Promoción a Caja Recaudadora (ProductoFinanciero)

El usuario confirma · se crea automáticamente un ProductoFinanciero tipo caja_efectivo con titularEntidadTipo: 'proveedor' · titularEntidadId: GK_id · ahora trackeable con saldo.

3 caminos para crear la caja · ⭐ B es el principal

Camino A Pre-creación manual

Operador setup-first · va a Tesorería → "Nuevo producto" → tipo "Caja recaudadora" → selecciona proveedor (GK) → elige uno de sus datosBancarios.

Ventaja: control total · creas la caja antes de que ocurra el primer cobro.
Desventaja: requiere recordar hacerlo · más clicks.
Útil para operadores estructurados que prefieren preparar todo antes.
⭐ Camino B Detección automática

Operador registra cobro normal · tipea/pega el número del yape · el sistema detecta que es de GK (de su datosBancarios) · banner aparece sugiriendo promover.

Ventaja: menor fricción · 1 click para confirmar · el sistema trabaja por ti.
Desventaja: requiere que datosBancarios estén bien registrados antes.
Camino principal recomendado · flujo natural del operador.
Camino C Selector inline

En el PagoUnificadoForm, opción "Yape (de tercero)" muestra un selector con todas las cajas recaudadoras existentes + opción "Crear nueva caja".

Ventaja: elección explícita · útil cuando hay muchas cajas (5+).
Desventaja: más fricción visual · pasos extra.
Útil cuando ya tienes varias cajas activas y necesitas elegir.

⚡ Las discusiones se resuelven viendo saldos por método separados · cero ambigüedad.
