Lo que se aprende del Lote 4

El split aplica en los 4 modos. Lo que cambia es: (1) qué grupos se ven en el selector de origen, (2) si hay selector de destino o no, (3) qué tercero está pre-cargado. Ese es el "scope contextual" del componente.

4

API completa · Props + Result + Migración backwards-compatible

El componente actual (v2 · 663 líneas · 25+ usos en el sistema) sigue funcionando sin tocar nada después del refactor. Las features nuevas se activan via props opt-in. Cero breaking changes.

PagoUnificadoFormProps
Interfaz de entrada del componente
// Props existentes en v2 · NO cambian
interface PagoUnificadoFormProps {
  isOpen: boolean;
  onClose: () => void;
  onConfirm: (result: PagoUnificadoResult) => Promise<void>;

  modo: 'cobro' | 'pago';
  saldoMaximo: number;
  monedaSaldo: 'PEN' | 'USD';
  docOrigenId?: string;
  docOrigenLabel?: string;

  // Props NUEVAS v3 · todas opt-in (default false)
  permiteSplit?: boolean;             // FASE 3
  permiteDestinoTercero?: boolean;   // FASE 2
  permiteAdjuntos?: boolean;         // FASE 3

  proveedorId?: string;             // FASE 2 · pre-carga tercero
  clienteId?: string;
  colaboradorId?: string;

  // Scope contextual del selector
  scopeOrigen?: Array<
    'mis-cuentas' |
    'cajas-recaudadoras' |
    'yapes-pasivos'
  >;
  scopeDestino?: Array<
    'datos-tercero' |
    'sin-destino-digital'
  >;
}
PagoUnificadoResult
Resultado entregado a onConfirm()
interface PagoUnificadoResult {
  // Campos legacy v2 · NO cambian (1-a-1 sigue OK)
  cuentaOrigenId: string;
  metodoPago: string;
  montoOriginal: number;
  monedaOrigen: 'PEN' | 'USD';
  tipoCambio?: number;
  fecha: Date;
  referencia?: string;
  notas?: string;

  // Campos NUEVOS v3 · opcionales
  cuentaDestinoId?: string;       // FASE 2

  // Si hay split: array de líneas
  // Si presente, los campos de arriba son del primer item
  lineas?: Array<{          // FASE 3
    cuentaOrigenId: string;
    cuentaDestinoId?: string;
    metodoPago: string;
    monto: number;
    moneda: 'PEN' | 'USD';
    comprobanteId?: string;
  }>;

  comprobantes?: Array<{        // FASE 3
    id: string;
    nombreArchivo: string;
    url: string;
    tamañoBytes: number;
  }>;
}

Fase 3 Flag PAGOFORM_V3_SPLIT=true · activar en flujos donde más urgente (caso GK Xpress en cobros de venta) · UAT con datos reales · luego rollout gradual al resto

Decisiones declaradas en este mockup

D-PAGOFORM-V3-1
1 componente · 4 modos · 0 breaking changes. v3 mantiene 100% de compatibilidad con v2 · todas las features nuevas son opt-in via props. Los 25+ usos actuales siguen renderizando igual.
D-PAGOFORM-V3-2
9 bloques · 5 siempre presentes · 4 condicionales. La complejidad visual escala según el contexto · gastos simples ven 5 bloques · pago a colaborador con split puede ver los 9. UI no se siente sobrecargada nunca.
D-PAGOFORM-V3-3
Selector ORIGEN con hasta 3 grupos jerárquicos: Mis cuentas (siempre) · Cajas recaudadoras (solo cobro) · Yapes pasivos (raro · solo cobro). Color violet para cajas · zinc para pasivos · resuelve la confusión actual de "¿dónde clasifico esto?".
D-PAGOFORM-V3-4
Selector DESTINO opt-in con 2 grupos: datos del tercero (referencia · NO mueve saldo) + sin destino digital (mano · agregar nuevo). Solo se muestra cuando permiteDestinoTercero=true Y hay tercero pre-cargado.
D-PAGOFORM-V3-5
Toggle Completo/Parcial/Split. Split es opt-in. Cuando está activo: 1 a N líneas con su propio origen + destino + monto · validación inline de totales · cada línea = 1 MovimientoFinanciero independiente vinculado al mismo doc origen.
D-PAGOFORM-V3-6
Preview de impacto antes de confirmar. Banking-grade: muestra antes/después de cada cuenta afectada + impacto en CTRU si modo='pago colaborador'. Reduce errores destructivos. Coordinado con D-INT-MOV-5 (impacto si elimino).
D-PAGOFORM-V3-7
Adjuntos múltiples opt-in · 1 voucher por línea de split. El voucher se asocia visualmente a su línea · queda como evidencia de auditoría · aporta valor especialmente en cobros split donde "le yapeé / pagué efectivo" puede generar discusiones.
D-PAGOFORM-V3-8
Rollout con feature flag por fase. Fase 1 sin flag (solo CSS) · Fase 2 con flag PAGOFORM_V3_DESTINO · Fase 3 con flag PAGOFORM_V3_SPLIT. Permite revertir a granular sin perder lo de antes.
D-PAGOFORM-V3-9
Esfuerzo total: ~3 sesiones de implementación. Fase 1 (3-4h) + Fase 2 (1 ses) + Fase 3 (1.5 ses). Cada fase autocontenida · si solo se completa Fase 1+2, sistema gana 70% del valor sin implementar split todavía.
