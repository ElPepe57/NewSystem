SECCIÓN 2 · Pack como FLAG · explicación con ejemplo visual concreto

¿Qué significa "Pack es un flag dentro de Skincare"?

Tu duda fue clara. Te lo muestro con un ejemplo concreto del modelo de datos.

Ejemplo 1 · Producto Skincare normal (NO es pack)
{ sku: "SKC-0042", nombreComercial: "Vitamin C Serum 30ml", marca: "SkinCeuticals", lineaNegocio: "skincare", ← LÍNEA esPack: false, ← FLAG OFF componentesPack: undefined, atributosSkincare: { tipoSKC: "serum", volumen: "30 ml", ... } }
Ejemplo 2 · Pack Skincare (sí es pack · MISMA línea SKC)
{ sku: "SKC-0099", nombreComercial: "Pack Anti-Edad Premium", marca: "SkinCeuticals", lineaNegocio: "skincare", ← MISMA línea esPack: true, ← FLAG ON componentesPack: [ { sku: "SKC-0042", nombre: "Vitamin C Serum", cantidad: 1 }, { sku: "SKC-0058", nombre: "Retinol Cream", cantidad: 1 }, { sku: "SKC-0071", nombre: "Eye Contour", cantidad: 1 } ], atributosSkincare: // heredados del componente principal }
Visual · pack es un producto SKC más + apunta a otros SKUs SKC

SKC-0058 · Retinol

El pack VIVE dentro de Skincare. NO es una línea aparte. NO se duplica el atributosSkincare. Solo apunta a 3 SKUs hijos existentes.
Implicaciones positivas
  • • Filtros por línea Skincare incluyen los packs (porque SON skincare)
  • • Reportes financieros agrupan packs y productos en mismo bucket
  • • Pack hereda categorías + etiquetas del componente principal
  • • Wizard Pack es WizardSimple + sección Componentes (no es 4to wizard separado)
Cuidados de implementación
  • • Card del pack DEBE mostrar avatar pack (gift) para distinguir visualmente
  • • Stock del pack es derivado del mínimo de componentes (no propio)
  • • Eliminar componente vinculado debe alertar al pack que lo usa
  • • Investigación + tools aplican a packs como cualquier producto SKC
Tu observación original
Tu frase: "los packs en realidad son parte de SKC, no entiendo porque figuran como una categoria extra, sino este jala todo lo que compone los productos del pack". Exactamente eso es lo que estoy modelando: el pack jala (apunta a) los productos del pack, vive dentro de SKC, no es una línea separada. La columna "Pack" en la tabla anterior fue confusa porque parecía ser una línea más. Ahora la mostramos como sub-flag de SKC.
SECCIÓN 3 · Contador usadoEn · qué es y cómo se ve en la UX

¿Qué es el contador "usadoEn" y para qué sirve?

Es un número que cada entrada de los catálogos creables guarda · cuenta cuántos productos lo usan actualmente.

Modelo de datos · ejemplo de la collection "marcas"
collection: marcas/ doc1: { id: "m-001", nombre: "SkinCeuticals", usadoEn: 24, ... } doc2: { id: "m-002", nombre: "The Ordinary", usadoEn: 18, ... } doc3: { id: "m-003", nombre: "Cosrx", usadoEn: 12, ... } doc4: { id: "m-004", nombre: "Sports Research",usadoEn: 8, ... } doc5: { id: "m-005", nombre: "Now Foods", usadoEn: 3, ... } doc6: { id: "m-006", nombre: "Brand X", usadoEn: 0, ... }
Cada vez que se crea un producto y se elige una marca, +1 al contador de esa marca. Cada vez que se elimina un producto, -1.

Mi pregunta original (decisión técnica)
Te pregunté: "¿agregamos el contador a los catálogos existentes en una migración, o solo lo activamos para los nuevos?"

Decisión recomendada: agregar `usadoEn` a TODOS los catálogos (existentes + nuevos) en una sola migración. Cloud Function que cuenta los productos por marcaId/tipoProductoId/etc y popula el contador. Opcional: mantener el contador actualizado vía Firestore Trigger en cada create/delete de producto.
