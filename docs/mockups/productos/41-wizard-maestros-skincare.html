Estado D · Arquitectura recomendada para escalar a 150+ productos por línea

Performance + gobernanza para 150+ productos por línea

Recomendación arquitectural para que el sistema escale sin perder velocidad de uso ni calidad de datos.

1 catálogo Firestore por atributo creable
  • • marcas/ · ya existe (proveedoresStore)
  • • tiposProducto/ · ya existe
  • • categorias/ · ya existe
  • • etiquetas/ · ya existe
  • • sabores/ · NUEVO para SUP
  • • restricciones/ · NUEVO para SUP
Atributos cerrados (Tipo SKC, Tipo piel, etc.) viven como ENUM hardcoded · NO requieren Firestore
Carga lazy + cache global
  • • Al abrir wizard: fetch paralelo de los 4 maestros
  • • Cache 10min en Zustand store global (SWR pattern)
  • • Subsiguiente apertura: ya cargado · 0ms
  • • Peso típico: marcas ~30 docs · ~50KB JSON
Para 150 prods × 10 atributos × promedio 8 valores únicos = 80 docs por catálogo · ínfimo en Firestore
Sugerencias por contador de uso
  • • Cada doc del catálogo tiene usadoEn: number
  • • Dropdown ordena DESC por usadoEn
  • • Más usados arriba · más fácil reuso
  • • Cuando creo producto, +1 al contador
Normalización + dedup en client
  • • Antes de guardar: trim().toLowerCase()
  • • Compara contra normalizado existente
  • • Si match → reusa el existente · NO duplica
  • • Display preserva mayúsculas/espacios originales
Gobernanza · módulo "Mantenimiento Maestros" (futuro)
Pantalla admin que muestra catálogos con duplicados detectados (ej: "Vit C" + "vitC" + "Vitamina C"). Admin elige cuál es la canónica y el sistema migra todas las referencias en 1 click. Beneficio: usuarios crean rápido sin friction · admin consolida después con visibilidad total.
