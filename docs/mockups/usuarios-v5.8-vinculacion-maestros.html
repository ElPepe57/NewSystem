¿Cuáles tienen "persona detrás"? · candidatos a vinculación con Usuarios
✅ Clientes

Si es persona natural (no empresa) · vincula con User externo subTipo=cliente_vip. Si es empresa B2B · vincula con su contacto humano.

✅ Proveedores

Cada proveedor tiene 0..N contactos (sales rep · contacto admin · contacto logístico). Cada contacto = User externo subTipo=contacto_proveedor.

⚠️ Marcas/Competidores

Si son empresas representadas por personas (ej. brand manager) · opcional. Por defecto NO vincular · es analytics no operacional.

⚠️ Lo que NO existe hoy en Maestros (decisión consciente)
❌ Transportistas

Olva · Shalom · couriers · motorizados freelance. Hoy se manejan ad-hoc en Envíos sin maestro. v5.8 propone: agregar como Maestro nuevo si se requiere · separado de esta vinculación.

❌ Instituciones financieras

BCP · BBVA · Mibanco. Hoy se manejan como CuentaCaja en Tesorería. Decisión: NO crear maestro · ya están como cuentas.

❌ Colaboradores marketing

Influencers · agencias. Hoy NO existen como maestro. v5.8 propone: viven SOLO en Users como externo · sin crear maestro propio.

ACTO 1 · Bloque A

Modelo conceptual · Maestros (entidad) ↔ Usuarios (persona)

Dos dominios distintos · vinculados por relación opcional. NO se fusionan · cada uno mantiene su rol.

DOMINIO 1
Maestros
Entidades comerciales / catálogos del negocio

Cosas con las que se opera · facturan · contabilizan. Empresas o personas naturales con RUC/DNI para documentos formales.

Características
  • · Documento legal: RUC · DNI · razón social
  • · Términos comerciales (crédito · pago)
  • · Historial transaccional (compras · ventas · OC)
  • · Sin acceso al sistema (no login)
  • · Vive en `/maestros`
clientes/{id}
proveedores/{id}
marcas/{id} · canalesVenta/{id}
DOMINIO 2
Usuarios
Personas físicas del ecosistema del negocio

Humanos con identidad · email · teléfono. Pueden tener acceso al sistema o no.

Características
  • · Documento personal: DNI · CE · pasaporte
  • · Email · teléfono · whatsapp
  • · Relación con el negocio (empleado · socio · honorarios · externo)
  • · Posible acceso al sistema (login)
  • · Vive en `/usuarios`
users/{uid}
relacionesLaborales/{id}
(modelo v5.6)
PUENTE · vinculación opcional bidireccional
Maestro
ej. proveedor Skin Labs SAC
proveedor.id
vinculado a
via campo entidadMaestroRef
Usuario · Externo
ej. Ricardo Castro · sales rep
relacion.tipo='externo'
Regla: la vinculación es 1..N · un Maestro puede tener N Users vinculados (contactos múltiples) · pero un User externo apunta a 0 o 1 Maestro (su empresa madre · si aplica).
ACTO 2 · Bloque A

Las 3 reglas de oro · cuándo crear qué

Matriz de decisión clara · evita duplicaciones · cada caso del negocio tiene UN patrón canónico.

📋
REGLA 1

Solo Maestro

Entidad comercial sin contacto humano recurrente que necesite gestión personalizada.

Cuándo:
  • · Proveedores transaccionales sin relación cercana
  • · Clientes de venta única / esporádicos
  • · Empresas grandes sin contacto único
👤
REGLA 2

Solo Usuario

Persona del staff o externo sin facturación formal vinculada a entidad comercial existente.

Cuándo:
  • · Empleado · socio · honorarios interno
  • · Colaborador puntual sin RUC
  • · Influencer marketing por canje
🔗
REGLA 3

Ambos vinculados

Cuando la persona física Y la entidad comercial son ambas relevantes para operar.

Cuándo:
  • · Sales rep de proveedor recurrente
  • · Cliente VIP persona natural
  • · Brand manager de marca aliada

interface RelacionLaboral {
  id: string;
  userId: string;
  tipo: 'empleado' | 'honorarios'
      | 'socio' | 'externo';
  estado: ...;
  fechaInicio: Timestamp;
  fechaFin?: Timestamp;
  motivoFin?: ...;
  datosLaboralesSnapshot?: ...;
  datosSocioSnapshot?: ...;
  datosExternoSnapshot?: ...;
  subTipo?: ...;
  creadoPor: string;
  fechaCreacion: Timestamp;
  notas?: string;

  // ✨ NUEVO v5.8 · solo para tipo='externo'
  entidadMaestroRef?: {
    tipo: 'cliente' | 'proveedor'
        | 'marca' | 'transportista';
    id: string;
    rolEnEntidad?: string;
  };
}
Lado Maestros · query derivado (NO se duplica data)
Cómo obtener los contactos de un Maestro:
// service · proveedoresService.getContactos(proveedorId)
const q = query(
  collection(db, 'relacionesLaborales'),
  where('tipo', '==', 'externo'),
  where('estado', '==', 'vigente'),
  where('entidadMaestroRef.tipo', '==', 'proveedor'),
  where('entidadMaestroRef.id', '==', proveedorId)
);
const snap = await getDocs(q);
const contactos = await Promise.all(
  snap.docs.map(d => userService.getUser(d.data().userId))
);
Ventaja del diseño:
  • ✓ 1 fuente de verdad: el vínculo vive en RelacionLaboral · NO se replica en Maestro
  • ✓ Sin migración brusca: campo opcional · viejos datos siguen siendo válidos
  • ✓ Reactivo: agregar/quitar contacto desde Maestros o Usuarios actualiza ambos lados
  • ✓ Cross-tipo: mismo modelo sirve para proveedor · cliente · marca · transportista
  • ✓ Validación Firestore: rules pueden verificar que entidadMaestroRef.id existe

⚠️ Riesgo · referencias colgadas

Si se borra un Proveedor desde Maestros mientras tiene Users vinculados, queda entidadMaestroRef apuntando a id inexistente. Mitigación: Firestore rule + Cloud Function que verifica y permite o re-asigna · soft delete recomendado.

⚠️ Riesgo · doble entrada

Admin podría crear un User externo y un Cliente por separado sin vincular. Mitigación: al crear User externo (subTipo cliente_vip), wizard sugiere buscar/crear en Maestros · al crear Cliente persona, ofrecer vinculación a User si existe.

⚠️ Riesgo · alcance creep

v5.8 puede tentar a agregar "Transportistas" como nuevo Maestro · feature creep. Mitigación: mantener alcance solo a Proveedores/Clientes/Marcas existentes · agregar Transportistas en sesión separada si se justifica.
