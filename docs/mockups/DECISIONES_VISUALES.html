⚙️ Decisiones de implementación

D-IMPL-1 · Orden de implementación

Mi propuesta de orden por valor operativo + dependencias visuales:

  1. Productos · tabla canónica (variante C robust grid) → uso diario, alta visibilidad
  2. Stock · listado canónico (variante C robust grid) → uso diario, mismo patrón que Productos
  3. Ventas · cards anchored (variante D + B híbrido) → entrada de ingresos del negocio
  4. Reclamos · timeline procedural → cierra workflow de incidencias
  5. Conversion + Transferencia wizards 3 pasos (variante B stepper)
  6. Pagos masivos wizard pixel-perfect (variante A sidebar vertical)
  7. Tarjeta crédito detalle + Caja recaudadora wizard
  8. Finanzas overview/cash-flow/modales (paquete completo)
  9. Tesorería layout shell + visualizaciones (separable)

D-IMPL-2 · Protocolo de validación previa

Para cualquier cambio grande/riesgoso (tabla compleja, wizard multi-paso, refactor estructural): te consulto antes con estimación de tiempo y opciones claras. Nunca decisión unilateral de "parchar".

D-IMPL-3 · Mockups viejos: ¿actualizar?

Los mockups Era 1-3 (rework-maestro · oc-detail-* · etc.) NO usan el lenguaje canónico final. Opciones:
(a) Actualizar mockups viejos al estándar nuevo antes de implementar
(b) Implementar al estándar nuevo aunque difiera del mockup viejo (mockup viejo queda como referencia histórica)
(c) Implementar tal cual el mockup viejo (acepta inconsistencia · no recomendado)
