Vista 4 · 8 gaps identificados · qué necesitamos vs qué tenemos

Necesidades del negocio (tus mensajes) que el componente actual no cubre.

GAP 1
Sin split por método
Tu caso: cliente paga S/500 yape + S/300 efectivo. Hoy hay que registrar 2 pagos separados, perdiendo el vínculo a la misma venta.
GAP 2
Sin destino "tercero" para cobros
Solo muestra cuentas propias en el selector. Si el cliente paga al yape de GK, no hay forma de seleccionarlo.
GAP 3
Sin lookup en datosBancarios
No detecta automáticamente que un yape pertenece a un proveedor (Red Logística), aunque esos datos ya están registrados.
GAP 4
Sin promoción a Caja Recaudadora
No hay banner sugiriendo "promover datoBancario a CuentaCaja" cuando aplica (el flujo S58c está pendiente).
GAP 5
Semántica "origen" para cobros
Internamente la prop se llama cuentaOrigenId aunque sea destino en cobros. Confusión técnica para mantenimiento.
GAP 6
Sin "agregar dato bancario inline"
Si quieres registrar un yape nuevo de un proveedor existente, hay que ir a su ficha · interrumpe el flujo del cobro.
GAP 7
Sin captura de comprobante
No permite adjuntar foto del yape, recibo de transferencia, etc. Para resolver discusiones después con GK falta evidencia adjunta.
GAP 8
Visual S58e no aplicado
El componente está en estilo legacy v2 · no usa el manifesto S58f (pill chips, tabular-nums, banking-grade cards).

Vista 5 · Propuesta v3 · cambios mínimos pero impactantes

Cómo evolucionar el componente sin romper los 25+ lugares que ya lo usan.

Estrategia · backwards compatible

Mantener la API actual intacta · agregar nuevas props/campos como opcionales · migración gradual módulo por módulo. 

// API v3 · backwards compatible

interface PagoUnificadoFormProps {
  // ... props actuales sin cambios ...

  // ─── NUEVAS PROPS (opcionales · opt-in) ───

  proveedorId?: string;                // contexto del proveedor (para lookup datosBancarios)
  permiteAdjuntos?: boolean;           // permite subir comprobante · default false

interface PagoUnificadoResult {
  // ... campos actuales sin cambios (1-a-1 sigue funcionando) ...

  // ─── NUEVO CAMPO OPCIONAL ───
  lineas?: Array<{                    // si presente · es split · ignora cuentaOrigenId/metodoPago
    cuentaId: string;
    metodoPago: string;
    monto: number;
    referencia?: string;
    comprobanteUrl?: string;
  }>;
}
