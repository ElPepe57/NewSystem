Alineación de color · Usuarios

A diferencia de Contabilidad (donde purple era pura decoración), en Usuarios el violet/purple significa 3 cosas a la vez. Este mockup muestra la madeja actual y propone una resolución limpia. Vos decidís antes de tocar código.

El problema · 3 usos mezclados + 1 inconsistencia

1 · Marco del módulo (purple)
breadcrumb · chip rol · icono header · tab activo · KPI "TOTAL" · filtros. Debería ser violet (Usuarios ∈ Equipo).
2 · Dato "socio" (¡2 colores!)
KPI "SOCIOS" sale violet en un strip y purple en otro. Inconsistencia real hoy.
3 · Rol "admin" (purple)
Parte del sistema de color por rol (admin=purple · gerente=indigo · vendedor=sky · finanzas=teal…). Es semántico.

La propuesta · separar identidad de dato

A · Marco (chrome) → violet. breadcrumb · chip · icono header · tab activo · filtros. Igual que Inversionistas y Planilla (todos Equipo).
B · "Socio" se unifica en violet en TODOS lados (KPI + tipo de relación). Coherente: socio = Inversionistas = Equipo = violet. Se arregla la inconsistencia.
C · KPI "TOTAL" → slate (neutral). Es un agregado general, no una categoría. Así no compite con "SOCIOS" violet en el mismo strip.
D · Sistema de color por ROL → intacto (semántico). admin se mantiene (badge pequeño, no es el marco). Único punto a confirmar abajo.

Sistema de color por rol · se mantiene (semántico)

Cada rol tiene su color para distinguirse en badges. Esto NO es chrome — es un sistema semántico de 9 categorías. Se conserva.

admin gerente vendedor comprador almacenero finanzas supervisor socio
El único punto a decidir: el badge del rol admin quedaría purple (es el único purple restante). Opciones: (a) dejarlo purple — es un badge pequeño, distinto del marco; (b) pasarlo a slate (admin = rol administrativo/root, neutro); (c) pasarlo a violet (pero chocaría con el rol "socio" que ya es violet).
